# v0.0.8 Installer UX Design

**Status:** APPROVED via brainstorm 2026-05-26
**Milestone:** v0.0.8-installer-ux (phased: v0.0.8a + v0.0.8b)
**Scope:** macOS only (Windows / Linux remain on legacy `install.sh` path; v0.0.7 W1 owns Windows finish)
**Repos in scope:** `smoodle` (this repo) + `smoodle-app` (sibling, Squirrel fork)
**Spec author:** founder (Apinant U-suwantim)

## Why v0.0.8 exists

Current install path (v0.0.6 / v0.0.7 macOS): user clones `smoodle`, runs `install-librime-fork.sh` (sudo, swaps dylib in upstream Squirrel), runs `install.sh` (copies schema YAMLs). Two scripts, two sudo prompts, fragile against Squirrel auto-updates wiping the dylib swap (CP-1).

v0.0.8 replaces the script path on macOS with a single drag-install DMG built from the `smoodle-app` Squirrel fork (`smoodle-type/smoodle-app`), which already exists at tag `v0.1.0` but ships an unpatched librime (caught during brainstorm 2026-05-26, fixed pre-spec; see § Pre-spec corrections). It also adds a `Smoodle Config.app` (Tauri) that surfaces three workflows: add personal words to the dict, check whether everything is healthy, change a few common settings.

## Pre-spec corrections (already landed)

During brainstorm, three latent bugs were surfaced and the most important one (librime patch lineage) was reconciled before this spec was written, so v0.0.8 starts from a verifiable foundation rather than a stale one:

- `smoodle-app/.gitmodules` pointed at `smoodle-type/librime` (correct, recent commit `7b587a2`) but the local submodule `.git/config` still pointed at `LoneExile/librime`. `git submodule sync` was never run after the URL update. The submodule was checked out at `1.16.1` from LoneExile — vanilla upstream, **no peek-sort patch**.
- The existing `dist/Smoodle-v0.1.0.dmg` was therefore built **without the patch** — the bug smoodle was created to fix was regressed in that DMG.
- Tag `1.16.0-smoodle.1` **does exist** on `smoodle-type/librime` (CLAUDE.md reference is correct; earlier `git ls-remote | head -10` truncated past it).

Reconciliation committed locally in `smoodle-app` as `1a0314c fix(librime): re-pin submodule to smoodle-type/librime@1.16.0-smoodle.1` (not pushed). Source is now correct; the existing v0.1.0 DMG remains stale and must be rebuilt before any v0.0.8 release.

## Decisions locked (do not relitigate without strong reason)

1. **v0.0.8 spans both repos** — `smoodle` ships Tauri Config app + schema source-of-truth; `smoodle-app` ships DMG + Sparkle infrastructure.
2. **Two `.app` bundles in one DMG** — `Smoodle.app` (pure IMK process, IME runtime) and `Smoodle Config.app` (Tauri, settings GUI). Independent updates, independent crash domains, independent signing scopes when v0.1.0 notarization arrives.
3. **Menubar shim** — `Smoodle.app`'s menubar gets an "Open Smoodle Config…" entry that fires `NSWorkspace.openApplication(at: /Applications/Smoodle Config.app)`. ~5 lines of Swift. Front door for the user; bundles stay separate.
4. **Schema via git submodule** — `smoodle-app` gets `vendor/smoodle/` as a submodule pinned to a smoodle tag. Build step copies `vendor/smoodle/schema/*.yaml` → `data/plum/`. CI has a drift-guard that fails if `data/plum/*.yaml` diverges from the submodule.
5. **Config app is Tauri 2 + Svelte 5** — no Tailwind. Vanilla CSS sized to match macOS native chrome. Universal arm64+x86_64.
6. **Three tabs locked: Words / Status / Settings** — content scope per § Config app feature set below.
7. **User-dict format = text YAML** — `~/Library/Rime/thai_phonetic.user.dict.yaml`. Standard Rime pattern. Git-friendly, survives reinstall, hand-editable as escape hatch.
8. **Auto-deploy on save** (toggle, default ON) — Config sends Apple Event "Deploy" to Smoodle.app via osascript after every Words / Settings edit. User can disable in Settings.
9. **Sparkle auto-update for both apps** — daily background check; `SUEnableAutomaticUpdates=false` so user approves each install. EdDSA-signed update payloads (`SPARKLE_PRIVATE_KEY` GHA secret).
10. **Hosting: GitHub Releases (DMG) + GitHub Pages (appcast.xml)** — appcast lives at `https://smoodle-type.github.io/smoodle-app/appcast.xml`. No new infra on dxc.0dl.me.
11. **Code-signing = unsigned for v0.0.8** — recruits see Gatekeeper warning, right-click → Open. Sparkle EdDSA still signs update payloads (separate from Apple signing). Notarization revisited at v0.1.0.
12. **Telemetry surface unchanged from v0.0.7** — install events from legacy `install.sh` keep firing; DMG install path does NOT emit telemetry. Documented gap. v0.0.9 may add runtime telemetry from Smoodle.app process itself.
13. **Phased delivery (Option B)** — v0.0.8a (~1.5 weeks) ships DMG + Sparkle only; v0.0.8b (~2 weeks after) adds Config app, delivered to v0.0.8a installs via Sparkle.

## Schema submodule mechanism

### Wiring

```
smoodle-app/
├── .gitmodules
│     [submodule "vendor/smoodle"]
│       path = vendor/smoodle
│       url = https://github.com/smoodle-type/smoodle.git
│       ignore = dirty
├── vendor/smoodle/                    ← submodule, pinned to TAG
│   └── schema/
│       ├── thai_phonetic.schema.yaml
│       ├── thai_phonetic.dict.yaml
│       └── default.custom.yaml
└── package/build-dmg.sh
      └─ step: cp vendor/smoodle/schema/*.yaml data/plum/
                (clobber any stale data/plum/*.yaml first)
```

### Lifecycle

```
smoodle repo:                      smoodle-app repo:
─────────────                      ─────────────────
1. founder edits schema/*.yaml
2. tag e.g. v0.0.8a-schema
3. git push --tags
                                   4. cd vendor/smoodle && git fetch
                                   5. git checkout v0.0.8a-schema
                                   6. cd .. && git add vendor/smoodle
                                   7. commit "bump schema to v0.0.8a-schema"
                                   8. tag smoodle-app v0.0.8a
                                   9. push → GHA builds DMG
```

### Drift prevention CI

Add to smoodle-app's `commit-ci.yml`:

```yaml
- name: schema-drift-guard
  run: |
    git submodule update --init vendor/smoodle
    for f in vendor/smoodle/schema/*.yaml; do
      base=$(basename "$f")
      if ! cmp -s "$f" "data/plum/$base"; then
        echo "DRIFT: data/plum/$base differs from vendor/smoodle/schema/$base"
        echo "       Run: cp vendor/smoodle/schema/*.yaml data/plum/ && commit"
        exit 1
      fi
    done
```

Same-class fix as the librime fiasco — make inconsistency a hard CI failure instead of silent staleness.

### Why submodule (not subtree, not fetch-at-build)

- **Submodule** — parent records exact SHA. `git log -- vendor/smoodle` shows discrete schema bumps. Reproducible builds at any commit.
- **Subtree** — schema copied into smoodle-app's history. Bigger repo, harder attribution, easier accidental edits to `data/plum/` (drift trap).
- **Fetch at build time** — breaks reproducibility (asset can be deleted/replaced), breaks offline builds, requires GH credentials in CI.

Submodule wins on auditability + offline reproducibility.

### Migration step (one-time, before v0.0.8a)

1. In `smoodle`: tag `v0.0.8a-schema` at current HEAD
2. In `smoodle-app`: `git submodule add https://github.com/smoodle-type/smoodle.git vendor/smoodle`
3. `cd vendor/smoodle && git checkout v0.0.8a-schema && cd ..`
4. Replace `data/plum/*.yaml` with `cp vendor/smoodle/schema/*.yaml data/plum/`
5. Add CI guard step
6. Commit `feat(schema): vendor smoodle as submodule (v0.0.8a-schema)`

user input  │                   │ stdlib  │                  │ filesystem

## Error handling

### Error taxonomy

| Failure | Where | User-visible behavior | Recovery |
|---|---|---|---|
| Smoodle.app not running | osascript deploy fails | Toast: "Smoodle not running — start it from /Applications, then click Re-deploy" + Re-deploy button | user double-clicks Smoodle.app; status bar flips green; Re-deploy works |
| Smoodle.app running but Apple Event not handled (older v0.0.8a build) | osascript returns AE timeout | Toast: "Deploy failed — your Smoodle.app needs update. Check menubar → Check for Updates." | Sparkle fetches v0.0.8b carrying the AE handler |
| `user.dict.yaml` write permission denied | Tauri command err | Modal: "Couldn't write to ~/Library/Rime/. Check Finder permissions or click 'Open Rime folder' to inspect." | escape hatch via Settings tab Open-folder button |
| `user.dict.yaml` malformed (hand-edited) | parse fails on `read_user_dict()` | Words tab: "Couldn't parse your user dict. [Open in editor] [Reset user dict (saves backup)]" | "Reset" writes `user.dict.yaml.broken-<ISO>` then writes empty header |
| Schema compile failed (deploy.log has `[error]`) | `schema_compile_log()` flags | Status tab red banner with last 5 error lines + [Open log] + [Reset to defaults] | Reset wipes `~/Library/Rime/*` except user.dict, re-copies bundled; user retries deploy |
| Sparkle download fails (network, GH 5xx, bad sig) | Sparkle handles | macOS dialog "Update failed. Will try again later." | next daily check; user can manual "Check for Updates" |
| Sparkle EdDSA mismatch | Sparkle blocks install | Modal: "Update rejected — signature invalid. Could be tampered download. Stay on current version." | manual report; founder rotates signing key |
| forget.0dl.me 401/403 (bad bearer) | `telemetry_forget()` err | Toast: "Couldn't delete server data — token rejected. Ask founder for new one." + [Skip server, wipe local only] | local files wiped regardless; server data persists until manual dxc cleanup |
| forget.0dl.me 502/network down | catch in Rust | Same toast with "Server unreachable. Try again later or [Skip server, wipe local only]." | local always wins; server delete retried next opt-in cycle |
| Apple Event sandbox prompt on first deploy (macOS automation permission) | macOS modal | User clicks Allow → permission persists; Deny → osascript fails -1743 | "App needs permission to control Smoodle: System Settings → Privacy & Security → Automation → Smoodle Config → enable Smoodle" instructions |

### Invariants enforced

- **`user.dict.yaml` never destroyed** — every write is read-modify-write with `.yaml.bak.<ISO>` left on failure. Reset-to-defaults explicitly preserves it.
- **`default.custom.yaml` never destroyed** — merge-patch (read YAML, mutate keys, serialize, atomic temp+rename). Bad patch → don't write.
- **No silent file overwrites** — pre-existing `~/Library/Rime/` files get `.smoodle-backup` on first run (same pattern install.sh uses).
- **Deploy failures don't roll back the file write** — YAML edit IS source of truth; deploy failure just means user hasn't seen it apply yet. They can click menubar → Deploy manually.

### Logging

Config.app keeps rolling log at `~/Library/Logs/Smoodle Config/config.log` (5 MB rotation, 5 files). Status tab has [View log] button → opens in Console.app. Every Tauri command logs `[ts] cmd=<name> ok=Y/N duration_ms=<n>` — debuggable by recruit without sending us anything.

No telemetry on errors (out of v0.0.8 scope; v0.0.9 if we want it).

### Out of v0.0.8 error scope

- Config.app crash recovery (Tauri/WebKit segfaults) — system handles, no internal restart loop
- Two Config.app instances editing user.dict.yaml simultaneously — Tauri single-instance plugin prevents
- Concurrent edit from external editor while Config.app open — last writer wins; document as limitation
- Smoodle.app / Config.app version skew — fallback: if osascript times out 3x, toast asks user to update Smoodle.app first

## Additional locked decisions (resolved during spec self-review)

14. **`data/plum/*.yaml` is `.gitignore`d in smoodle-app and regenerated at build time** — drift impossible by construction. Build step is the only source. Submodule is single source of truth. Migration step is therefore: `git rm --cached data/plum/*.yaml`, add to `.gitignore`, regenerate via `cp vendor/smoodle/schema/*.yaml data/plum/` as a build step. Drift-prevention CI guard from earlier section becomes a verification that `data/plum/` matches submodule AFTER the build step runs (not a comparison of tracked-vs-submodule, since the tracked version is gone).
15. **Tauri Config app does NOT auto-create `~/.smoodle/` on first launch** — only creates when user explicitly opts in to telemetry via the Status tab toggle. Aligns with "default-OFF, no PII without consent" privacy claim. First-launch shows Status tab with "Telemetry: off (opt-in required)" and no `install_id` exists until the user toggles on.
